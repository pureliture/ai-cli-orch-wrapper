# Milestone v1.0: ai-cli-orch-wrapper

**Status:** SHIPPED 2026-03-25
**Phases:** 1-3
**Total Plans:** 10

## Overview

v1.0 shipped the core portability promise of the wrapper: a fresh machine can be brought into the same AI CLI orchestration environment with one setup command, aliases and role mappings are configurable from repo-local config, and a real CAO-backed plan→review loop now runs end-to-end through repo-local artifacts.

## Phases

### Phase 1: Foundation + Environment Setup

**Goal**: Users can bootstrap a complete AI CLI orchestration environment on any machine with a single command
**Depends on**: Nothing (first phase)
**Plans**: 2 plans

Plans:

- [x] 01-01 — Delete PoC files, rewrite `src/cli.ts`, create `src/commands/setup.ts` and test scaffold
- [x] 01-02 — Build, run tests, and smoke-test `wrapper setup` on a real machine

**Details:**

- `wrapper setup` validates prerequisites, writes `~/.config/tmux/ai-cli.conf`, and injects a single `source-file` line into `~/.tmux.conf`
- setup is idempotent and was verified both by automated tests and live-machine smoke checks

### Phase 2: CLI Aliases + Workflow Config

**Goal**: Users can invoke AI CLIs via short wrapper aliases and declare role→CLI mappings in a config file without touching source code
**Depends on**: Phase 1
**Plans**: 4 plans

Plans:

- [x] 02-01 — Create RED test scaffolds for config and alias behavior
- [x] 02-02 — Implement `readWrapperConfig()` and alias dispatch
- [x] 02-03 — Wire dynamic alias dispatch into `src/cli.ts` and commit `.wrapper.json`
- [x] 02-04 — Build, run tests, and smoke-test live alias dispatch on a real machine

**Details:**

- `.wrapper.json` became the repo-local source of truth for aliases, roles, and later workflows
- built-ins stay ahead of aliases in dispatch order so config cannot shadow `setup`, `help`, or `version`
- live alias dispatch through CAO was verified on the developer machine

### Phase 3: Plan→Review Orchestration Loop

**Goal**: Users can run a structured plan→review workflow where two AI CLIs iterate until the reviewer approves or the iteration limit is reached
**Depends on**: Phase 2
**Plans**: 4 plans

Plans:

- [x] 03-01 — Lock workflow config resolution and strict `review.status.json` parsing
- [x] 03-02 — Lock repo-local artifact layout and planner/reviewer prompt contract
- [x] 03-03 — Implement the CAO execution seam and shared planner→reviewer loop engine
- [x] 03-04 — Expose the workflow loop through the CLI and verify it end-to-end

**Details:**

- workflow definitions resolve logical roles to providers through `.wrapper.json`
- run artifacts live under `.wrapper/workflows/<workflow>/runs/<run-id>/iterations/<nn>/`
- planner/reviewer approval is determined only from machine-readable `review.status.json`
- live named and ad-hoc workflow runs were smoke-tested successfully against CAO-backed provider sessions
- runtime hardening during verification changed completion logic to wait for artifacts on disk instead of trusting transient terminal states

## Milestone Summary

**Key Accomplishments**

- `wrapper setup` now bootstraps the environment in a single idempotent command
- alias and role mappings are configurable from repo-local `.wrapper.json`
- workflow definitions, reviewer status parsing, and artifact layout are fully locked by tests
- CAO HTTP session management and planner→reviewer looping are implemented in the wrapper
- `workflow` and `workflow-run` both completed real approval runs with preserved artifacts

**Key Decisions**

- Keep wrapper config minimal and avoid inventing a parallel workflow DSL
- Treat workflow artifacts on disk as the authoritative completion signal in live runs
- Use repo-local state and artifacts for full portability across machines

**Issues Resolved**

- Early CAO terminal state changes could incorrectly look "complete" before `plan.md` or review artifacts existed
- Real providers explored too much before writing required files, so prompts and timeout defaults were hardened during live smoke testing

**Issues Deferred**

- Milestone audit artifact was not produced for v1.0
- workmux-driven isolated worktree orchestration remains next-milestone work

**Technical Debt**

- Real provider runs are still relatively slow and depend on first-party CLI behavior inside tmux panes
- A first-run skills enablement prompt may still appear once in some provider sessions before becoming non-issue
