## Context

`aco`에서 provider 호출은 현재 스트리밍 기반으로 동작해 큰 출력 스트림을 직접 터미널로 넘기는 구조를 갖습니다.
그러나 과거 이슈 트랙에서 `fullOutput += chunk` 형태의 누적 버퍼링이 OOM의 원인이 될 수 있다는 지적이 있었고, 호출자별 출력 처리 정책이 명시적으로 분리되지 않으면 동일한 위험이 재발할 수 있습니다.

현재 코드 기준으로도 호출 경로를 통일해 옵션 기반의 출력 정책을 도입하면, 향후 `aco run`/`aco ask`/테스트 훅에서 요구되는 메모리 사용량을 일관되게 제어할 수 있습니다.

## Goals / Non-Goals

**Goals:**

- provider invoke 경로에 출력 버퍼링 정책(`stream-only`, `bounded`, `disabled`)을 추가한다.
- 기본 동작은 메모리 사용량이 제한된 안전 모드로 유지한다.
- 호출자가 필요할 때만 제한된 크기 버퍼를 사용하도록 API 계약을 제공한다.

**Non-Goals:**

- provider 출력 형식/콘텐츠 의미론 변경(필터링, 요약, 변환).
- 표준 출력 파이프라인의 기존 터미널 스트리밍 동작을 근본적으로 바꾸거나, 로그 저장 경로를 교체하는 행위.

## Decisions

1. **버퍼링 정책을 `spawn-stream` 호출 옵션으로 수렴**
   - 대안 A: 모든 호출에서 무제한 버퍼링으로 기본값 유지 -> 메모리 폭증 가능성 높음.
   - 대안 B: 디폴트로 `stream-only`, 필요 시 `bounded` 또는 `disabled` 선택 가능 -> 메모리 안정성과 유연성 균형.
   - 선택 이유: 안전 기본값을 제공하면서 기존 구현 패턴(스트리밍 루프)과 충돌이 적습니다.

2. **`bounded` 기본 크기 정책 설정**
   - 대안 A: 호출자가 명시적으로만 크기 지정 -> 설정 누락 시 동작 불명확.
   - 대안 B: 기본 상한(예: 1MiB) + 최대치 상한값 제공 -> 예측 가능한 메모리 사용량 보장.
   - 선택 이유: 실수/누락에도 방어적으로 동작하며 테스트 용이성이 높습니다.

3. **기존 호출자 기본값 변경 없이 점진 롤아웃**
   - 대안 A: 모든 호출을 한 번에 변경 -> 회귀 리스크 증가.
   - 대안 B: `aco run`, `aco ask` 호출에만 안전 정책을 먼저 적용하고, 필요 시 추가 호출자로 확장 -> 롤백 부담 감소.
   - 선택 이유: 리뷰/테스트 범위를 최소화하면서 핵심 OOM 경로를 선제적으로 차단할 수 있습니다.

## Risks / Trade-offs

- [Risk] 제한 버퍼로 인해 일부 디버그 유틸이 기대한 전체 출력 문자열을 못 받음 -> [Mitigation] bounded 모드의 한계를 문서화하고, full 버퍼링은 호출자 opt-in으로만 허용.
- [Risk] 정책 미설정 시 기본값 혼동 -> [Mitigation] 기본값은 `stream-only`로 명시하고, `bounded` 요청 시만 메모리 캡이 적용됨을 로그/주석으로 기록.
- [Risk] 구현 초기 테스트 부재 시 실제 메모리 절감 효과를 검증 실패 -> [Mitigation] 대용량 스트림 fixture를 포함한 회귀 테스트를 tasks에 포함.

## Migration Plan

1. provider invoke 옵션에 출력 정책 인터페이스만 추가하고, 기존 호출은 기본값(`stream-only`)으로 동작하게 유지한다.
2. `aco run`/`aco ask` 호출부에서 기본 동작을 안전 정책으로 정렬한다.
3. 필요한 경우에만 새로운 `bounded`/`disabled` 모드를 사용하는 코드를 점진 추가한다.
4. 문서/사용 예시를 갱신해 호출자가 메모리-안전 경로를 쉽게 선택할 수 있게 한다.
5. 성능/회귀를 확인한 뒤 장기적으로 정책 사용 지표(옵션 선택 빈도)를 추가 로그로 검토한다.

롤백은 기본 옵션 동작을 제거하지 않고 정책 입력값만 기본값으로 되돌리는 방식으로 수행한다.

## Open Questions

- `bounded`의 기본 상한(예: 1MiB)과 최대 상한(예: 16MiB)을 무엇으로 둘지에 대한 합의가 필요합니다.
- `disabled` 모드에서 호출자는 어떤 대체 데이터(예: status only)만 기대해야 하는지 정책 문서에서 명시해야 합니다.
