## Context

The current context sync implementation discovers `.claude/skills/*/SKILL.md` and plans a recursive copy for every discovered skill into `.agents/skills/<skill>/`. That behavior was useful when ACO treated Claude configuration as the only source of truth, but it now conflicts with provider-native and upstream-managed surfaces:

- Gemini already exposes ACO-owned `gh-*` entrypoints through `.gemini/commands/gh-*.toml`.
- ACO also has thin Codex command-alias skills under `.agents/skills/gh-*`, which Gemini can discover as skills and expose alongside its commands.
- OpenSpec and Superpowers install and update their own skills/plugins. Copying those assets into `.agents/skills`, `.codex/skills`, or `.gemini/commands` creates stale forks that ACO does not own.
- The sync manifest currently tracks only source and target hashes. It cannot distinguish ACO-owned generated assets from external assets that must be ignored.

This design narrows ACO's responsibility: ACO sync owns provider bootstrap and ACO-maintained policy/reference assets, while external tools own their own command and skill surfaces.

## Goals / Non-Goals

**Goals:**

- Make shared skill sync default-deny and allow only explicitly shared ACO-owned skills.
- Preserve recursive copying for allowed shared policy/reference skills, including bundled scripts and references.
- Classify OpenSpec, Superpowers, marketplace/plugin skills, and command-alias skills as external or provider-specific rather than shared outputs.
- Detect duplicate command/skill exposure across provider surfaces and report actionable warnings.
- Record ownership, kind, source, action, and warnings in `.aco/sync-manifest.json`.
- Provide a safe migration path for duplicate assets that were generated by earlier ACO behavior.
- Keep provider-specific commands on provider-specific surfaces.

**Non-Goals:**

- Reimplement OpenSpec or Superpowers installation.
- Replace Codex, Gemini, or Claude native skill discovery.
- Fork upstream OpenSpec/Superpowers prompts into ACO templates.
- Delete external upstream-owned source assets such as `.claude/skills/openspec-*`.
- Finalize a new Codex-only `$gh-*` command alias design in this change.

## Decisions

### 1. Classify Assets Before Planning Outputs

**Decision:** Source discovery remains broad, but sync planning SHALL classify each skill or command-like asset before creating outputs. The classifier returns an ownership decision such as `aco`, `external`, `provider-specific`, or `unknown`, plus a kind such as `shared-skill`, `command-alias-skill`, `external-skill`, or `provider-command`.

**Rationale:** Discovery should still see assets so `aco sync --check` can warn about duplicates, but output planning must not copy assets unless ACO owns them.

**Alternative considered:** Stop discovering external assets entirely. Rejected because diagnostics and migration need to see already-installed external assets.

### 2. Use Explicit Allow Rules for Shared Skills

**Decision:** A skill is eligible for `.agents/skills` only when it is explicitly allowed by `.aco/sync.yaml` or by `x-aco-owned: true` frontmatter. `.aco/sync.yaml` takes precedence over frontmatter, and `skills.exclude` has higher precedence than `skills.include`.

Default deny patterns include `openspec-*`, `superpowers-*`, `using-superpowers`, `brainstorming`, `writing-plans`, `executing-plans`, and `gh-*`.

**Rationale:** The safe default is to avoid copying assets ACO does not own. Frontmatter supports self-describing ACO-maintained skills, while repo-level config lets maintainers override policy centrally.

**Alternative considered:** Maintain a hardcoded allowlist only. Rejected because ACO-owned shared skills may grow and need repo-local configuration without code changes.

### 3. Treat Command Alias Skills as Provider-Specific

**Decision:** `.agents/skills` SHALL NOT be used as a command alias registry. Thin `gh-*` skills are not shared policy skills and SHALL be removed from shared sync outputs. Claude commands stay in `.claude/commands/`, Gemini commands stay in `.gemini/commands/`, and any future Codex `$gh-*` UX must use a Codex-only surface.

**Rationale:** Gemini can discover `.agents/skills`, so a command-alias skill there competes with `.gemini/commands/<name>.toml`.

**Alternative considered:** Keep `gh-*` in `.agents/skills` and teach Gemini to ignore them. Rejected because ACO should not rely on provider-specific negative discovery behavior for a shared directory.

### 4. Make Manifest Ownership Explicit

**Decision:** The manifest SHALL evolve from hash-only tracking to an ownership-aware model. It SHALL retain enough compatibility to read older manifests, but newly written manifests SHALL include per-target records with owner, kind, source, provider when relevant, action, hashes, and warnings.

External assets SHALL be recorded as ignored or skipped records, but SHALL NOT be overwritten by `aco sync` or `aco sync --force`. Adoption into ACO ownership requires an explicit future option such as `--adopt`; this change only reserves the ownership boundary and does not silently adopt external assets.

**Rationale:** Hashes alone cannot answer whether a file is generated output, a provider-native command, or upstream-managed external content.

**Alternative considered:** Encode ownership only in filename conventions. Rejected because collisions are exactly the failure mode this change addresses.

### 5. Add Provider Exposure Duplicate Detection

**Decision:** `aco sync --check` and sync planning SHALL build a provider exposure index from provider-specific commands and discovered skill aliases. The duplicate detector SHALL warn when the same provider can expose the same command or skill name from more than one surface.

For Gemini, an entry such as `.gemini/commands/gh-issue.toml` and `.agents/skills/gh-issue/SKILL.md` is a duplicate risk. For OpenSpec, duplicated `opsx` commands and `openspec-*` skills across Gemini/Codex/shared surfaces are reported as external duplicate assets with cleanup guidance.

`--strict` or CI mode SHALL promote duplicate warnings to command failure without writing files.

**Rationale:** Duplicate exposure is a usability and maintenance issue even when file contents are not identical. It needs a first-class diagnostic separate from content drift.

**Alternative considered:** Detect only exact path collisions. Rejected because the problem is provider-visible name collision, not filesystem path collision.

### 6. Cleanup Is Conservative

**Decision:** Cleanup planning SHALL remove duplicate generated assets automatically only when the manifest records them as ACO-owned and their current hash still matches the manifest. Unknown, unmanifested, or user-modified duplicate assets SHALL require an explicit force-clean option and SHALL be reported before deletion.

External source assets such as `.claude/skills/openspec-*` and `.claude/skills/superpowers-*` SHALL never be cleanup targets.

**Rationale:** This repository already contains generated and external-looking assets. Cleanup must avoid deleting user-installed upstream assets or hand-edited files.

**Alternative considered:** Delete all matching duplicate globs. Rejected because it would be unsafe in user repositories.

### 7. Pack Setup Owns Only the ACO Pack

**Decision:** `aco pack install` and `aco pack setup` SHALL install and validate ACO-owned command pack assets only. The post-setup sync step uses the same ownership classifier, so setup cannot spread OpenSpec or Superpowers assets into provider targets. `aco pack status` SHALL report ACO command pack state separately from external integration observations.

**Rationale:** Provider readiness and external integration installation are different concerns. ACO should not mark OpenSpec/Superpowers as required provider readiness.

**Alternative considered:** Make pack setup install OpenSpec and Superpowers adapters. Rejected because those tools own their own upstream install/update lifecycle.

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| Existing users rely on `.agents/skills/gh-*` for Codex `$gh-*` UX | Document the behavior change and defer replacement to a Codex-only alias design |
| External asset classifier misses a third-party skill | Default deny prevents accidental copying; unknown assets are reported rather than synced |
| Duplicate detector reports noisy warnings | Include cause files, provider name, exposed name, and recommended action so warnings are actionable |
| Manifest format migration breaks older generated outputs | Read legacy `sourceHashes`/`targetHashes` and write ownership records on the next successful sync |
| Cleanup deletes user edits | Auto-delete only manifest-owned matching hashes; require explicit force-clean for ambiguous files |

## Migration Plan

1. Add the duplicate detector in `aco sync --check` so current duplicates are visible before behavior changes remove outputs.
2. Add shared skill allow/exclude classification and manifest ownership records.
3. Stop planning `.agents/skills/gh-*`, `.agents/skills/openspec-*`, and `.agents/skills/superpowers-*` outputs.
4. Add conservative cleanup for manifest-owned duplicate outputs, with warnings for ambiguous duplicates.
5. Remove or document removal of ACO-generated `.gemini/commands/opsx/` and `.codex/skills/openspec-*` copies.
6. Update context sync, architecture, and root instruction docs to describe external asset ownership.
7. Validate with targeted sync tests, fixture tests, `aco sync --check`, and `git diff --check`.

### External Duplicate Asset Cleanup

The following directories contain upstream-managed external assets that were previously generated by ACO sync. They are not ACO-owned and should not be recreated by `aco sync`:

- `.gemini/commands/opsx/` — OpenSpec commands managed by the OpenSpec CLI. These are upstream-managed external integrations.
- `.codex/skills/openspec-*` — OpenSpec skills managed by the OpenSpec CLI. These are upstream-managed external integrations.

These directories may be removed manually if they are no longer needed, or left in place if they are maintained by external tools. `aco sync` will not create, update, or remove them automatically.

Rollback is file-based: keep the new classifier disabled behind the old sync path only during development, or restore the previous sync behavior from git. Once released, rollback should preserve manifest ownership records and avoid re-copying external assets.
