# Blueprint: v2.0 Multi-AI CLI Command Bridge

**Objective:** `ai-cli-orch-wrapper`를 cao 기반 오케스트레이션에서 gemini-cli / copilot-cli를 위한 ccg-workflow 스타일 커맨드 브릿지로 완전 피벗한다.
**Repo:** `ai-cli-orch-wrapper` (branch: `main`)
**Created:** 2026-04-01
**Status:** REVIEWED

---

## Context

### 이전 상태 (v1.1 — Shipped)
- `aco` CLI surface 고정 완료 (`aco` as canonical bin name)
- `.wrapper.json` 런타임 컨트랙트 확립 (`aliases`, `roles` 스키마)
- cao (AWS Labs CLI Agent Orchestrator)에 의존하는 plan→review 루프
- `src/orchestration/`: cao-client, workflow-runner, prompts, status-file, workflow-config
- `src/commands/`: alias, setup, workflow, workflow-run
- `src/config/aco-config.ts`: AcoConfig {aliases, roles, workflows}

### 피벗 이유
- cao 의존성 제거 결정: 외부 서버 없이 로컬 CLI dispatch 방식으로 전환
- codex는 OpenAI 플러그인으로 공식 지원 → 별도 브릿지 불필요
- gemini-cli / copilot-cli는 아직 플러그인/커맨드 브릿지 없음 → 직접 구현 대상
- 참조 아키텍처: `ccg-workflow` (라우팅 설정) + `codex-plugin-cc` (커맨드셋)

### 새 아키텍처 개요
```
aco init          → 라우팅 설정 (어떤 커맨드를 어떤 CLI로)
aco review        → 지정 CLI에 review 프롬프트 dispatch
aco adversarial   → 지정 CLI에 adversarial review dispatch
aco rescue        → 지정 CLI에 rescue 프롬프트 dispatch
aco status        → 지정 CLI 실행 가능 여부 + 현재 설정 확인
```

### 기술 스택 (변경 없음)
- TypeScript 5.x, Node.js >=18, zero runtime deps
- `src/` → `dist/`, `node --test` 테스트 러너
- `child_process.spawn` (async) — spawn 패턴은 **async streaming** 으로 고정
- Adapter dispatch 결과 타입: `{ stdout: string; stderr: string; exitCode: number }`

#### 핵심 설계 결정

**Config 스키마 (`.wrapper.json` 파일명 유지 — 하위 호환):**

```json
    "gemini": { "bin": "gemini", "args": [] },
    "copilot": { "bin": "gh", "args": ["copilot", "suggest", "-t", "shell"] }
  }
}
```

**CliAdapter interface (TypeScript):**
```typescript
export type AdapterErrorKind =
  | 'NOT_INSTALLED'
  | 'NOT_AUTHENTICATED'
  | 'TIMEOUT'
  | 'SPAWN_ERROR'
  | 'UNKNOWN';

export interface DispatchResult {
  stdout: string;
  stderr: string;
  exitCode: number;
}

export interface CliAdapterError extends Error {
  kind: AdapterErrorKind;
  adapter: string;
}

export interface CliAdapter {
  name: string;
  isAvailable(): Promise<boolean>;      // 빠른 가용성 확인 (cache 없음, 매번 확인)
  dispatch(
    prompt: string,
    opts?: { timeoutMs?: number }
  ): Promise<DispatchResult>;           // throw CliAdapterError on failure
}
```

**v1.x → v2.0 마이그레이션 규칙:**
| v1.x 필드 | v2.0 변환 | 규칙 |
|-----------|-----------|------|
| 없음 (version 필드 없거나 "1.x") | `version: "2.0"` 추가 | 항상 |
| `roles.reviewer` | `routing.adversarial` | reviewer → adversarial |
| `roles.orchestrator` | `routing.review` | orchestrator → review |
| `aliases` | 제거 (v2.0에서 별도 지원 안 함) | 경고 메시지 출력 |
| `workflows` | 제거 | 경고 메시지 출력 |

### Step 7 — `/rescue` Command

**Branch:** `feat/v2-rescue`
**Depends on:** Step 6
**Model tier:** default
**시간 예상:** 60분

#### Context Brief
`aco rescue`: stuck 상황을 탈출하는 명령. 에러 로그, 상황 설명을 넘기면 구체적 해결책을 제시한다.

**Input 우선순위 (rescue):**
1. `--from <file>` — 파일 내용을 컨텍스트로 사용
2. `--error <message>` — 직접 메시지
3. stdin (파이프) — `echo "error" | aco rescue`
4. 아무 것도 없으면 오류 출력

> stdin이 TTY가 아닐 때(파이프) 자동으로 stdin 모드 진입.
> `--from` + `--error` 동시 제공 시 둘 다 합쳐서 사용 (우선순위 아님, 병합).

#### Tasks
- [ ] `src/prompts/rescue.ts` — rescue 프롬프트 템플릿:
  ```typescript
  export function buildRescuePrompt(context: string): string {
    return `I am stuck. Here is the situation:

${sanitizeInput(context)}

Recent git context:
${sanitizeInput(recentGitLog)}

Provide concrete, actionable steps to unblock me.
Be direct. No disclaimers. No caveats.`;
  }
  ```
- [ ] `src/commands/rescue.ts` 구현:
  - `--from <file>` (파일 없으면 오류)
  - `--error <message>`
