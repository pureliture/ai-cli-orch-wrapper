Ubiquitous Language로 gray box의 회색을 선명하게 만든다

이 문서는 introduce-ubiquitous-language proposal을 구현 착수자가 빠르게 이해하도록 재구성한 HTML kick-off입니다. 핵심은 코드만이 아니라 용어, 문서, artifact 이름까지 같은 경계 언어로 맞추는 것입니다.

대상: maintainer와 AI agent 범위: docs first 금지: public CLI rename
한 문장 요약

흩어진 도메인 용어를 하나의 repo-owned glossary로 묶어 aco ask, aco run, aco sync가 같은 의미 체계에서 계획되고 검증되게 만든다.

요약 Gray Box 시각화 용어 지도 영향 범위 착수 질문 원문

01. Executive Summary

proposal의 핵심은 세 가지입니다

1

언어도 interface다

현재 용어는 README, architecture, security, tests, implementation에 흩어져 있습니다. 문서 작성과 갱신을 LLM이 보조하더라도 최종 기준은 repo 안에 남아야 하며, agent가 매번 의미를 다시 추론하면 gray box 경계가 흐려집니다.

2

repo-owned glossary를 만든다

docs/reference/ubiquitous-language.md 를 source of truth로 두고, provider invocation, session artifact, context sync, harness boundary 용어를 정렬합니다.

3

작게 연결하고 drift를 막는다

모든 문서를 한 번에 다시 쓰지 않습니다. 용어가 처음 등장하는 문서에 링크를 추가하고, 처음부터 자동 terminology check를 붙여 LLM-assisted 문서 갱신의 drift를 잡습니다.

02. Boundary Model

gray box에서 “회색”이 의미하는 것

회색은 완전한 black box도, 완전한 white box도 아닙니다. 내부 구현을 모두 노출하지 않더라도, maintainer와 agent가 같은 용어로 입력, 출력, evidence, artifact를 추적할 수 있어야 합니다.

SOURCE SURFACE

repo가 관리하고 LLM이 보조하는 기준

  • README와 architecture docs
  • security와 session artifact reference
  • LLM-assisted docs와 review notes
  • OpenSpec capabilities와 scenarios
  • public CLI contract
GRAY BOX LANGUAGE LAYER

공통 언어가 통과시키는 것

  • 동일한 의미의 run과 session
  • provider output을 advisory로 다루는 규칙
  • artifact와 brief의 책임 분리
  • harness와 generated target의 경계
GENERATED TARGET

agent가 생성하거나 수정하는 표면

  • provider prompts와 review notes
  • tests와 emitted artifact names
  • future OpenSpec changes
  • terminology-sensitive runtime output

03. Term Map

먼저 정렬해야 할 용어 묶음

Provider Invocation

외부 AI CLI를 호출하고 결과를 advisory evidence로 다루는 영역입니다.

provider advisory permission profile aco ask aco run

Session Artifacts

실행 결과가 사람이 검토 가능한 증거로 남는 영역입니다.

run session artifact brief output log

Context Sync

repo source와 generated target이 어떻게 연결되는지 설명하는 영역입니다.

source surface generated target sync manifest managed block drift

Harness Boundary

agent와 wrapper가 어디까지 책임지는지 구분하는 영역입니다.

harness wrapper CLI contract review gate verification

04. Scope and Impact

바뀌는 곳과 바꾸지 않는 곳

New

새 기준 문서

docs/reference/ubiquitous-language.md 를 glossary와 naming rules의 기준점으로 추가합니다.

Touch

기존 문서 링크

README, architecture, security, session artifact docs에서 용어가 처음 등장하는 지점에 짧은 설명이나 링크를 붙입니다.

Guard

가벼운 drift 방지

broad rename 대신 초기 자동 terminology check와 review checklist를 함께 두고, 충돌 용어를 조기에 잡습니다.

Non-goal

공개 contract 보존

aco CLI 이름 변경, 스타일성 대규모 rename, 전 문서 일괄 rewrite는 이번 범위가 아닙니다.

05. Kick-off Questions

구현 세션을 시작할 때 적용할 결정

1차 glossary 범위

좁게 시작합니다. 1차 범위는 provider invocation과 session artifacts로 제한하고, 나머지 용어는 placeholder와 backlog로 남겨 2차 확장에서 다룹니다.

자동 check 시작점

처음부터 자동 terminology check를 넣습니다. PR review checklist는 보조 채널로 두고, false positive는 alias나 allowlist로 조정합니다.

code rename 강도

너무 엄격한 rename campaign은 피합니다. public contract는 보존하되 혼란을 줄이는 targeted naming, alias, docs-only 정렬을 우선합니다.
